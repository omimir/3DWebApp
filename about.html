About this project

Vitrine is a small 3D web app that shows three models I built in Blender: a drinks can, a water bottle and a desk globe. This page covers how the models were made, why the app is put together the way it is, how the code is organised, and what I did for accessibility.

The 3D models

The can and bottle are each a little over sixteen thousand vertices, with their curves carried by smooth shading rather than extra polygons. The globe's sphere is the densest mesh of the three, since a round surface needs the geometry to stay round close up.

The materials are physically based, set up with Blender's Principled shader and exported as glTF metallic-roughness. Each model has an image texture for its artwork: printed labels on the can and bottle, a world map on the globe. The viewer relights everything with an image-based environment and a studio light rig, so the metal can, the clear plastic bottle and the matte globe all catch the light in their own way.

The app ignores the lights and cameras that come inside the glTF file. They are stripped out as the model loads, and the app adds its own light rig and its own perspective and orthographic cameras. That way it keeps full control of how each model is lit and framed, and both can be changed from the controls.

Design choices

The interface uses a dark palette so the lit models are the brightest thing on the screen. The layout is a fluid Bootstrap grid. On a wide screen the gallery, the viewer and the controls sit side by side; on a narrow screen they stack into one column so the viewer stays a usable size on a phone.

The controls sit in the right-hand panel: wireframe, a spotlight, the globe's surface texture, and the post-processing toggle. Each model also has an animation of its own behind the Animate button. The can's tab pops open, the bottle's cap unscrews and lifts away, and the globe spins on its stand and slowly coasts to a stop. A separate Explode button pulls each model apart into its pieces.

Rendering happens on demand. The scene is only redrawn when something actually changes, like the camera moving or an animation running, so the page sits quiet while you are just reading. That keeps it light on the processor and the battery.

The three models are also provided in X3D, a declarative 3D format. Those were converted from the glTF exports and live in a separate repository. I picked those three specifically because I've always loved globe displays, having wanted to be an astronomer as a kid. As for the water bottle, I just found Evian's mountain design to be cool, and the can was to one-up the one shown in the labs. I hope I did that justice.

Architecture: model, view, controller

The code under js/ is split into model, view and controller, written as plain JavaScript ES modules with no build step.

The model holds the state. AppState tracks what the app is currently showing: the active model, wireframe, lighting, camera mode, animation and effects. ModelCatalog holds the list of models, fetched at runtime from data/models.json with the fetch API, which is where the project uses AJAX and JSON. Any time the state changes, it announces the change on a small publish-and-subscribe EventBus.

The views listen to that bus and never change state themselves. SceneView owns the Three.js renderer, scene, cameras and render loop. GalleryView, ControlPanelView, InfoPanelView and TextureSwapView draw the rest of the interface from the same model data and redraw when the bus tells them to.

The controllers turn input into state changes. UIController handles clicks on the gallery and the controls, RaycastController handles clicks on the 3D model itself, KeyboardController handles arrow-key navigation, and AppController connects state changes through to the scene. Input only flows one way: a controller changes the model, the model announces it, the views react.

So the stack is HTML5, CSS3 and JavaScript, with Three.js doing the 3D and Bootstrap doing the responsive layout, and the content loaded as JSON over fetch.

Deeper understanding: shaders and post-processing

The globe has a thin extra shell around it running a custom GLSL ShaderMaterial. Its fragment shader works out a fresnel term: bright where the surface turns away from the camera, fading where it faces you. That falloff reads as a faint atmosphere at the globe's edge.

The viewer also has an optional post-processing chain built with the Three.js EffectComposer: the scene render, a bloom pass on the bright highlights, tone mapping, a second piece of custom GLSL for a vignette that darkens the frame edges like a studio photo, and an anti-aliasing pass. The bloom is eased back if your system asks for reduced motion.

Accessibility

The pages use semantic HTML5 landmarks (header, nav, main, footer), and every control is a real button, so you can reach the whole app with the Tab key. The model gallery also responds to the arrow keys, and focus follows whatever is selected.

Toggle buttons report their state with aria-pressed. The model description sits in an aria-live region, so a screen reader announces it when it changes. The controls keep plain text labels and a visible focus outline.

Text stays well above the WCAG AA contrast ratio against the dark background. If your system asks for reduced motion, the exploded view and the model animations skip their tweens and settle straight to the end state, and the bloom is eased back. The WebGL canvas has a text fallback for browsers that cannot show it.
